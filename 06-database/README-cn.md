# 第六章：数据库（不是背 SQL，是理解“为什么它这么快/这么稳”）📚

> 这一章的目标不是“会写 SELECT”，而是：  
> 当系统慢了、数据错了、并发上来了、线上炸了——你能从上帝视角判断问题在哪一层，以及下一步该怎么查。  
> AI 可以帮你写 SQL，但 AI 很难替你背锅：性能、正确性、事务、锁、连接池这些坑，最终还是你要懂。🙂

本章按你的 `outline.md` 分成五个阶段：

- 第一阶段：CRUD / Join / Group / Index（先会用，再知道为什么）
- 第二阶段：B+Tree / Hash Index / Clustered Index / Covering Index（索引的底层逻辑）
- 第三阶段：事务 / ACID / MVCC / Lock / Deadlock（正确性与并发）
- 第四阶段：Explain / 执行计划 / 慢查询 / 优化（让数据库“做对题”）
- 第五阶段：连接池 / Pool 的意义 / 连接为何不能无限 / 泄露 / 复用（工程落地）

## 按三个问题拆解

### 1) 为什么会出现数据库？🤔

如果没有数据库，你的系统很快会变成这样：

- 数据存在内存：机器重启=全丢
- 数据存在文件：能存但很难查，更新也痛苦
- 数据存在“各种 JSON”：早晚变成“谁都不敢改”的屎山

现实世界需要的不是“能存”，而是同时要：

- **能长期保存**（持久化）
- **能快速查询**（索引、执行计划）
- **能并发访问**（锁、MVCC、隔离级别）
- **能保证正确性**（事务、约束）
- **能恢复和审计**（日志、WAL、备份）

数据库就是把这些需求打包成一个“靠谱的事实来源（Source of Truth）”。

### 2) 为什么选关系型数据库？有没有别的选择？🧰

关系型数据库（MySQL/PostgreSQL）适合：

- 强结构化数据（表、字段）
- 关联关系（用户-订单-商品）
- 事务（钱、库存、权限）
- 复杂查询（筛选、排序、聚合）

替代方案也很多，但各有代价：

- KV（如 Redis）：快，但不擅长复杂查询和强事务持久化
- 文档库（如 MongoDB）：结构灵活，但事务、复杂 JOIN 思维不同
- 列存/OLAP（如 ClickHouse）：分析强，但不适合高频事务写
- 搜索引擎（如 Elasticsearch）：检索强，但不是事实来源

大白话：  
**关系型数据库是“总账本”，其他很多系统是“加速/检索/分析工具”。**

### 3) 数据库到底怎么工作？我们怎么用？🔍

下面开始“把数据库掀开盖子看”。

## 你必须先建立的一个直觉：数据库慢，通常不是因为它“算不过来”

数据库慢，常见原因不是 CPU 不够，而是：

> **磁盘/内存访问模式不友好** + **没走对索引** + **执行计划选错路** + **锁冲突** + **连接用法错误**。

所以你需要把数据库想成一个“超级会管理 IO 的系统”。

先来一张手绘图理解它最关心的东西：I/O。

```text
CPU 计算：很快（纳秒~微秒）
内存访问：很快（纳秒）
磁盘访问：很慢（毫秒级）

数据库的主要目标之一：
尽量用内存 / 顺序读 / 少随机 IO
```

## 第一阶段：CRUD / Join / Group / Index（先会用）

你先不用写出天花乱坠的 SQL，你要会写“可读、可维护、能走索引”的 SQL。

### 1. CRUD：最基本但最常见

```sql
-- Create
INSERT INTO users (id, email) VALUES (1, 'a@b.com');

-- Read
SELECT id, email FROM users WHERE id = 1;

-- Update
UPDATE users SET email = 'new@b.com' WHERE id = 1;

-- Delete
DELETE FROM users WHERE id = 1;
```

你要建立的直觉：

- `WHERE` 是“过滤条件”，通常决定能否走索引
- `SELECT *` 在生产里经常是坏味道（多取数据、影响覆盖索引）

### 2. Join：关系型数据库的灵魂技能

假设有两张表：

```text
users(id, email)
orders(id, user_id, total)
```

查询“用户 + 订单”：

```sql
SELECT u.id, u.email, o.id AS order_id, o.total
FROM users u
JOIN orders o ON o.user_id = u.id
WHERE u.id = 1;
```

手绘图理解 JOIN：

```text
users(1)  ----(u.id = o.user_id)----> orders(很多条)
```

新手常见坑：

- 忘了 `ON` 条件导致笛卡尔积（行数爆炸）
- JOIN 的列没有索引导致全表扫描 + 大量比较

### 3. Group By：把数据“归堆”

统计每个用户订单数：

```sql
SELECT user_id, COUNT(*) AS cnt
FROM orders
GROUP BY user_id;
```

直觉：

- `GROUP BY` 通常需要排序/哈希聚合，数据量大时很吃资源
- 统计类查询很多时候更适合走“分析库/报表系统”，但业务里也经常用

### 4. Index：数据库加速的“地图”

你可以先把索引理解成：

> 让数据库不用翻完整本书，而是用目录直接跳到页码。

```text
没有索引：一本书从第一页翻到最后一页找一句话
有索引：先看目录 -> 跳到对应页
```

创建索引的直觉：

- 常用过滤条件（WHERE）列适合建索引
- 高选择性（区分度高）的列更适合（如 user_id 通常比 gender 更适合）

## 第二阶段：索引为什么快（B+Tree / Hash / Cluster / Cover）

这一节是“数据库为什么快”的核心。

### 1. B+Tree：为什么大多数索引长这样？

数据库最怕随机 IO。

B+Tree 的设计非常适合“磁盘/页”这种存储模型：

- 树高度很低（通常 3~4 层就能放很多数据）
- 叶子节点按顺序链接（范围查询很爽）
- 每一层节点可以对应“页”，减少 IO 次数

手绘一个 B+Tree 的感觉：

```text
            [  50  ]
           /       \
      [10 20 30]  [60 70 80]
        | | |        | | |
        v v v        v v v
      (叶子节点按顺序链起来)
  [1..9] <-> [10..19] <-> [20..29] <-> ...
```

你要记住它的两个“工程价值”：

- **等值查询**：WHERE id = 123 很快
- **范围查询**：WHERE time BETWEEN ... 也很快（叶子有序）

### 2. Hash Index：为什么它不是默认主角？

Hash 索引擅长等值查询：

```text
key -> hash(key) -> bucket -> record
```

但它不擅长范围查询：

- 你想查 100~200 的记录，Hash 不知道“范围”是什么

所以很多关系型数据库默认主力还是 B+Tree。

### 3. Clustered Index：为什么“数据本体”也会被索引组织？

某些数据库/存储引擎会让数据按主键顺序组织存储（聚簇）。

直觉：

- **数据行本身按主键排序**，主键范围查询很舒服
- 但主键如果乱（比如 UUID 随机），插入会更碎，更容易产生页分裂

手绘理解：

```text
聚簇：数据本体按主键顺序放
页1: [id 1..100]
页2: [id 101..200]
...
```

### 4. Covering Index：为什么只查索引就能返回结果？

如果你的查询只需要的列都在索引里，数据库可以：

> 只读索引，不回表读数据行。

例子：

```sql
-- 假设有索引 (user_id, created_at, total)
SELECT user_id, created_at, total
FROM orders
WHERE user_id = 1
ORDER BY created_at DESC
LIMIT 20;
```

如果 `SELECT` 的列都在索引里，就可能走覆盖索引，少一次 I/O。

你要建立的直觉：

- `SELECT *` 会毁掉很多“只走索引”的机会
- 只取需要的列，既省网络也省磁盘

## 第三阶段：事务（ACID / MVCC / Lock / Deadlock）

这一节是“数据库为什么稳”的核心。

### 1. 事务是为了解决什么？

你最常见的业务动作：

> 扣库存 + 生成订单 + 扣款

如果没有事务，你可能出现：

- 订单写了，库存没扣
- 库存扣了，订单没写
- 扣款成功，订单失败

事务就是为了让一组操作：

- 要么都成功
- 要么都失败

### 2. ACID：别背字母，记住人话

- A 原子性：要么全做，要么全不做
- C 一致性：做完后系统仍符合规则（约束、外键、业务不变量）
- I 隔离性：并发时互不干扰到“读到诡异状态”
- D 持久性：提交后不会因为宕机就消失

### 3. 并发下会出现哪些“诡异现象”？

你需要知道这些词背后的故事：

- 脏读：读到了别人未提交的数据
- 不可重复读：同一查询两次结果不一样（别人改了并提交）
- 幻读：同一条件两次查询，出现“多/少了几行”（别人插入/删除）

手绘时间线：

```text
时间 ->

事务A: 读余额=100 -------------- 再读余额=80
事务B:          更新余额=80 并提交

如果隔离不够：A 两次读不一致（不可重复读）
```

### 4. MVCC：为什么读不一定要加锁？

MVCC（多版本并发控制）可以让“读”看到一个一致快照：

- 写入会产生新版本
- 读可以读旧版本（取决于隔离级别）

手绘理解版本链：

```text
Row X:
版本1 (v1) -> 版本2 (v2) -> 版本3 (v3)

读：拿到某个时间点的快照 -> 读对应版本
写：产生新版本
```

直觉：

> MVCC 是让“读变便宜”的关键之一。

### 5. Lock：为什么还需要锁？

不是所有事都能靠 MVCC 解决。

尤其是写写冲突、某些隔离级别要求时，锁仍然是必需的。

常见锁概念（先记直觉，不用背细节）：

- 行锁：锁某一行
- 表锁：锁整张表
- 共享锁（读锁）：多人可读
- 排他锁（写锁）：写时别人别碰

### 6. Deadlock：为什么会死锁？

死锁的本质是：

> A 等 B 手里的锁，B 等 A 手里的锁，大家一起“互相礼让到天荒地老”。😵

手绘：

```text
事务A: 锁住行1 -> 想锁行2（被 B 占着）
事务B: 锁住行2 -> 想锁行1（被 A 占着）
=> 死锁
```

工程经验：

- 数据库通常会检测死锁并“杀掉其中一个事务”
- 你要学会：业务层重试 + 统一加锁顺序（降低死锁概率）

## 第四阶段：Explain / 执行计划 / 慢查询 / 优化

这一节是“你从上帝视角救火”的核心技能。

### 1. 为什么需要执行计划？

同一句 SQL，数据库可能有很多种执行方式：

- 先扫谁
- 用什么索引
- 怎么 JOIN
- 是否排序

执行计划就是数据库的“解题步骤”。

你要学会的不是背计划字段，而是学会问：

> 它为什么走了这条路？有没有更好的路？

### 2. EXPLAIN 的大白话读法

不同数据库输出不一样，但你先抓住核心：

- 有没有走索引
- 扫描了多少行（预估/实际）
- 有没有排序/临时表
- JOIN 的顺序是什么

手绘执行计划的意义：

```text
SQL: 我要查一条记录
数据库：我有两条路
  路A：全表扫描 1000万行
  路B：走索引，直达 1 行

EXPLAIN：告诉你它选了哪条路
```

### 3. 慢查询怎么排？

你可以用三步走（非常实用）：

1. **先确定慢在哪里**：是 DB 慢，还是应用慢，还是网络慢
2. **拿到慢 SQL**：看日志/慢查询
3. **用 EXPLAIN 观察“走的路”**：有没有索引，扫描行数是否离谱

### 4. 最常见的优化方向（按性价比排序）

- 只查需要的列（别 `SELECT *`）
- 增加合适索引（尤其是 WHERE/JOIN/ORDER BY 相关）
- 避免在索引列上做函数/隐式转换（容易导致不走索引）
- 控制返回行数（LIMIT）
- 避免 N+1 查询（应用层循环查）
- 把复杂查询拆为两步（先筛 ids，再回表取详情）

## 第五阶段：连接池（Pool 的意义 / 为什么不能无限 / 泄露 / 复用）

### 1. 为什么需要连接池？

很多新手以为数据库连接像 HTTP 请求一样：

> 用完就关，下次再开。

现实是：数据库连接很贵。

- 建连接要握手、鉴权、分配资源
- 连接数过多会让数据库上下文切换压力暴增

所以连接池的意义是：

> **复用少量连接服务大量请求**，并且给连接数一个上限。

### 2. 为什么连接不能无限创建？

因为数据库不是无限大的“电话客服中心”：

- 每个连接都要占内存
- 每个连接都要占 CPU（管理）
- 并发太高时，系统可能会把自己管理死

所以工程上必须：

- 限制连接数
- 用队列排队
- 给超时兜底

手绘图理解连接池：

```text
请求进来
   |
   v
[连接池]
   | 有空闲连接 -> 借给你用
   | 没空闲连接 -> 排队等待 / 超时失败
   v
[数据库]
```

### 3. 连接泄露是什么？

连接泄露 = 你借了连接不还。

结果：

- 池里的连接越来越少
- 最后所有请求都卡在“等连接”

典型原因：

- 异常路径没有释放连接
- 代码提前 return 没释放

### 4. 用 Node 写一个“连接池思维”的伪代码

这里不绑定具体库，你抓住模式就行：

```js
async function handler(req, res) {
  const client = await pool.acquire();
  try {
    const result = await client.query("SELECT ...");
    res.end(JSON.stringify(result));
  } finally {
    pool.release(client);
  }
}
```

注意 `finally`：这是防泄露的关键。

## AI 时代你最该掌握的数据库能力（上帝视角版）🧠

你不需要把每个实现细节背下来，但你必须具备这些“判断力”：

1. **数据模型与约束意识**：什么数据是事实来源，什么是缓存/派生数据
2. **索引直觉**：为什么这条查询走索引/不走索引，怎么设计组合索引
3. **事务语义**：你的业务能接受什么隔离级别，哪里必须强一致
4. **执行计划意识**：性能问题第一反应是 EXPLAIN，而不是“先加机器”
5. **连接池与容量意识**：数据库不是无限，连接数、慢查询、锁冲突会互相放大

如果你能把这些建立起来，你写系统设计/做排障/做性能优化就有底气了。

## 加餐 1：数据建模（表怎么设计才不坑未来的自己）🧱

很多数据库事故，根源不是 SQL 写错，而是“模型一开始就歪了”。

你需要的不是花里胡哨的范式理论，而是几个非常实用的工程原则。

### 1) 先画出“事实”和“关系”，再写表

大白话：

- 事实（facts）：必须被长期保存的真实记录（订单、支付、出入库）
- 关系（relations）：事实之间如何关联（user_id、order_id、product_id）

手绘：

```text
用户(User) ----< 订单(Order) ----< 订单项(OrderItem) >---- 商品(Product)
```

如果你能先把这张关系图画对，后面 70% 的坑会直接消失。

### 2) 约束不是“限制你”，是“保护你”

新手常见心态：

> “约束好烦，我先不加，后面再说。”

然后后面就会出现：

- 重复数据
- 脏数据
- 外键断裂
- 业务对不上账

你至少要习惯这几类约束的存在感：

- `NOT NULL`：这字段到底能不能没有？
- `UNIQUE`：邮箱、用户名、订单号能不能重复？
- 外键（可选）：关系是否需要数据库层保证？

### 3) 规范化 vs 反规范化：不是对错，是取舍

你可以把它理解成：

- 规范化：少冗余，更新更安全，但查询可能需要 JOIN
- 反规范化：查询更快更方便，但更新时需要同步多份数据

经验法则：

- 事实数据（订单、支付）尽量规范化，避免对账灾难
- 展示数据（商品详情页）可以反规范化，但要有一致性策略

### 4) 常见字段设计坑（你以后一定会踩）

- 金额用 `float/double`：会出现精度问题（钱不要用浮点）
- 时间字段混乱：本地时区/UTC 没统一
- 字符集/排序规则不一致：导致比较和索引行为诡异
- 软删除不设计：或者设计了但所有查询忘记过滤

## 加餐 2：主键怎么选？自增 vs UUID（它会影响性能）🆔

### 1) 自增主键（递增 ID）

优点：

- 写入更顺序，页分裂少
- 索引更紧凑
- 范围扫描更舒服

缺点：

- 容易暴露数据规模（猜 ID）
- 分库分表场景需要额外方案

### 2) UUID 主键（随机）

优点：

- 全局唯一，分布式方便
- 不容易被猜

缺点（这是重点）：

- 随机写入导致 B+Tree 插入更“碎”
- 更容易页分裂，索引膨胀

### 3) 常见折中：有序 UUID / 雪花 ID

目标是：

> 既全局唯一，又尽量递增，兼顾分布式和写入友好。

你不需要一上来就实现雪花算法，但要知道为什么很多公司会搞这个。

## 加餐 3：分页为什么经常慢？Offset 分页 vs 游标分页 📄

### 1) Offset 分页（最常见也最容易慢）

```sql
SELECT id, created_at
FROM orders
ORDER BY created_at DESC
LIMIT 20 OFFSET 100000;
```

问题是：数据库往往需要“走过前面 100000 行”，再返回 20 行。

越翻越慢。

### 2) 游标分页（Keyset Pagination，生产更常用）

思路：

> 不要告诉数据库“跳过多少行”，而是告诉它“从哪个位置继续往后”。

```sql
SELECT id, created_at
FROM orders
WHERE created_at < ?
ORDER BY created_at DESC
LIMIT 20;
```

手绘：

```text
Offset：翻到第 100000 行再开始拿
游标：从上次最后一条的 created_at 继续往后拿
```

## 加餐 4：N+1 查询：数据库性能杀手（而且很隐蔽）🕳️

现象：

- 你查了一批订单（1 次 SQL）
- 然后循环对每个订单查用户（N 次 SQL）
- 结果总共 N+1 次查询

伪代码：

```text
orders = SELECT * FROM orders LIMIT 100
for order in orders:
  user = SELECT * FROM users WHERE id = order.user_id
```

解决思路：

- 用 JOIN 一次查完
- 或者 IN 查询批量拉取（再在应用层组装）

```sql
SELECT * FROM users WHERE id IN (...100 个 id...);
```

## 加餐 5：读写分离与主从复制（为什么会“刚写完查不到”？）🔁

你在 outline 的“下一章问题”提到了复制延迟，这里先给你上帝视角版本。

### 1) 为什么要主从？

- 主库写压力大
- 读请求更多

所以：

- 写去主库
- 读尽量去从库

### 2) 复制延迟是怎么来的？

因为复制需要时间：

```text
主库写入 -> 写入日志 -> 传输 -> 从库应用日志 -> 从库可读
```

中间任意一段慢了，就会出现：

> 你刚写完，读从库读不到（短暂）。

### 3) 工程上怎么处理？

常见手段：

- 写后读一致：刚写完的一段时间强制读主库
- 业务容忍最终一致：部分页面允许稍旧
- 把“关键链路”保持在主库读

## 加餐 6：慢查询排障清单（线上救火版）🧯

当你遇到“数据库慢”的报警，别先改代码，先按流程走：

### 0) 先确认：慢的是 DB 还是别的？

最常见的误判：

- 应用线程池/事件循环卡住，但你以为是 DB
- 网络抖动，但你以为是 SQL

### 1) 找出最慢的 SQL（别猜）

- 应用日志
- 慢查询日志
- APM/Tracing（如果有）

### 2) EXPLAIN 看三件事

- 有没有走索引
- 扫描行数是否离谱
- 有没有排序/临时表

### 3) 再看三件系统级指标

- 连接数：是不是打满了
- 锁等待：是不是在等锁
- I/O：是不是磁盘在喘

### 4) 最高 ROI 的临时止血策略

- 限流/降级（保护 DB）
- 加索引（谨慎，注意在线 DDL 成本）
- 调整查询只取必要列
- 拆查询（先查 id，再查详情）

## 加餐 7：备份与恢复（你迟早会被问到）🧰

备份不是“有就行”，你必须回答这两个问题：

1. **我能恢复吗？**（恢复演练）
2. **我能恢复到多新？**（RPO：最多能丢多少数据）

你不需要在这里学全套命令，但你要有这几个概念：

- 全量备份 vs 增量备份
- 备份频率
- 恢复演练（比备份更重要）
- 权限与加密（备份文件本身也是敏感数据）

## 加餐 8：ORM 的正确打开方式（别把 ORM 当成魔法）🧙

ORM 很方便，但它也很容易把你带进坑里：

- 不知不觉 N+1
- 不知不觉查太多列
- 不知不觉事务范围太大
- 不知不觉把索引优化空间吃掉

一个成熟的工程习惯是：

> 你可以用 ORM 写业务，但你必须能读懂它生成的 SQL，并能用 EXPLAIN 验证它的执行路径。
