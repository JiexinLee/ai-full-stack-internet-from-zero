# 第八章：并发（前端最容易忽略，但一上生产就躲不开）⚙️

> 你的 `outline.md` 说得很真实：很多前端对并发几乎没概念。  
> 但并发是“系统能不能扛住”的基础之一。  
> AI 可以帮你写代码，但并发问题通常是：**程序跑着跑着就慢了/卡了/死锁了/把 CPU 打满了**。  
> 这些问题需要你有模型、有直觉、有排查路径。

本章按你的提纲来：

- Process / Thread / Coroutine / Async / Event Loop
- 然后：线程池 / 对象池 / 连接池 / Worker / Cluster
- CPU 密集 vs IO 密集
- 为什么 Java 有线程池、Node 没有（或看起来没有）
- Node 为什么不用线程？libuv 干什么？
- 为什么图片压缩不能放主线程？

---

## 按三个问题拆解

### 1) 为什么会出现并发问题？🤔

因为现实世界的请求不是排队一个个来的：

- 用户同时点
- 定时任务同时跑
- IO 同时返回
- CPU 同时被抢

如果你的系统只会“一个一个处理”，你会看到：

- 延迟暴涨
- 吞吐下降
- 请求排队像春运

所以系统必须学会并发：

> **同时处理多个任务，或者让等待 IO 的时间不白等。**

### 2) 为什么选择这些并发模型？有没有别的选择？🧰

常见并发模型有几类：

- 多进程：隔离强，但成本高
- 多线程：共享内存但需要锁，复杂度高
- 协程/异步：把“等待”变成可调度，适合 IO 密集
- 事件循环：用单线程调度大量 IO

每种模型都在解决一个核心矛盾：

> **我们想同时处理很多事情，但 CPU 核心和资源是有限的。**

### 3) 并发到底怎么工作？开发里怎么用？🔍

从开发者视角，你最重要的是学会三件事：

1. 识别你的任务是 CPU 密集还是 IO 密集
2. 识别你当前语言/运行时的并发模型（特别是 Node）
3. 学会用正确的工具把“重活”从关键路径移走（worker/队列/池/异步）

---

## Part 0：先把“并发世界观”摆正 🧠

### 0.1 并发 vs 并行

很多人把这两个词混着用。

- 并发（concurrency）：看起来同时在做多件事（调度）
- 并行（parallelism）：真的同时做多件事（多个 CPU 核）

手绘：

```text
并发：一个人同时处理 3 个锅（不断切换）
并行：三个人各处理一个锅（同时进行）
```

### 0.2 你写的代码“快不快”，常常不是算法，而是等待

很多线上慢不是 CPU 算不过来，而是：

- 等数据库
- 等网络
- 等磁盘
- 等锁
- 等连接池

所以并发的核心价值之一是：

> **等待期间别闲着，让 CPU 去干别的。**

---

## Part 1：Process（进程）是什么？🏭

进程像一个独立工厂：

- 有自己的内存空间
- 彼此隔离
- 一个进程崩了，另一个不一定崩

手绘：

```text
进程A：内存A + 资源A
进程B：内存B + 资源B
（隔离）
```

代价：

- 创建/切换更贵
- 通信麻烦（IPC）

适合：

- 强隔离需求
- 多核并行（多个进程跑在不同 CPU 核上）

---

## Part 2：Thread（线程）是什么？🧵

线程像一个工厂里的工人：

- 同一个进程里的线程共享内存
- 共享意味着“沟通快”
- 也意味着“容易互相踩脚”（需要锁）

手绘：

```text
进程
  |
  +-- 线程1 共享内存
  +-- 线程2 共享内存
  +-- 线程3 共享内存
```

优点：

- 通信快（共享内存）
- 并行能力强（多核）

缺点：

- 线程安全问题（锁、竞态）
- 调试困难（偶现 bug）

---

## Part 3：Coroutine（协程）是什么？🧶

协程可以理解成：

> 更轻量的“可暂停/可恢复”的任务单元。

它通常不由 OS 直接调度，而是由语言/运行时调度。

直觉：

- 线程更像“系统级工人”
- 协程更像“应用级流程”

协程最擅长：

- IO 密集：等待网络/磁盘时切出去，让别的协程跑

---

## Part 4：Async（异步）到底在说什么？⏳

异步不是“更快”，异步是：

> **把等待变成可管理的状态**，让你在等待期间还能做别的事。

同步流程：

```text
做A -> 等A结束 -> 做B -> 等B结束
```

异步流程：

```text
发起A -> 先去做B -> A好了再回来处理
```

---

## Part 5：Event Loop（事件循环），Node 世界的交通指挥 🧑‍✈️

Node 的经典模型是“事件循环 + 非阻塞 IO”。

你可以把 event loop 想成一个单线程的调度员：

- 你把任务丢给它
- 它把 IO 交给系统/线程池
- IO 完成后再把回调放回队列
- 主线程负责把回调一个个执行掉

手绘图：

```text
主线程（Event Loop）
   |
   +-- 发起 IO（网络/磁盘）
   |
   +-- 继续处理其他回调
   |
   +-- IO 完成 -> 回调入队 -> 主线程执行
```

大白话总结：

> Node 擅长同时处理很多“会等待 IO 的任务”。  
> Node 不擅长在主线程做很重的 CPU 计算。

---

## Part 6：CPU 密集 vs IO 密集（你做架构判断的第一把尺子）📏

### 6.1 CPU 密集

特点：

- 主要花时间在计算上
- CPU 使用率高
- 单线程会被占满

例子：

- 图片压缩
- 视频转码
- 大量加密/解密
- 大规模 JSON 解析/序列化

### 6.2 IO 密集

特点：

- 主要花时间在等待上（网络/磁盘/DB）
- CPU 可能并不高

例子：

- 调数据库
- 调第三方 API
- 读写文件（大量小文件）

手绘快速判断：

```text
CPU 密集：CPU 在忙
IO 密集：CPU 在等
```

---

## Part 7：为什么 Java 有线程池？为什么 Node “看起来没有”？🏊

### 7.1 Java 的世界观（简化版）

Java 服务常用多线程处理请求：

- 每个请求占用一个线程（或线程池中的线程）
- 线程阻塞等待 IO 时，这个线程就“挂起等”

为了避免无限创建线程，必须有线程池：

> **线程很贵，不能无限开。**

### 7.2 Node 的世界观（简化版）

Node 的主线程不想被阻塞。

它用 event loop 处理大量并发连接，IO 交给系统去做。

所以 Node 常说“单线程”，意思更接近：

> JavaScript 代码执行在一个主线程上（event loop）。

但并不意味着 Node 完全没有线程。

---

## Part 8：libuv 到底干什么？🧰

libuv 是 Node 的底层 IO 库，主要负责：

- event loop
- 异步 IO 抽象
- 线程池（处理一些阻塞型任务）

你可以先记住一句话：

> Node 的“异步能力”，很多时候靠 libuv 把脏活累活接走。

比如某些文件 IO、DNS 查询、加密相关工作，可能会落到 libuv 的线程池中处理。

---

## Part 9：为什么图片压缩不能放主线程？📸

因为它是典型 CPU 密集任务。

如果你在 Node 主线程里做 CPU 重活：

- event loop 被卡住
- 所有请求都得排队
- 用户体验像“全站卡死”

用一张图理解：

```text
主线程在压缩图片（CPU 重活）
   |
   v
其他请求的回调都排队等着
   |
   v
延迟暴涨
```

所以 CPU 重活通常要：

- 放到 worker 线程
- 或者交给独立服务/队列系统

---

## Part 10：Worker / Cluster：Node 怎么“用上多核”？🧑‍🔧

### 10.1 Worker Threads：把 CPU 重活挪走

Node 有 `worker_threads` 可以把 CPU 密集任务放到工作线程。

一个最小示例（主线程发消息，worker 算完回消息）：

`main.js`：

```js
import { Worker } from "node:worker_threads";

function runWorker(input) {
  return new Promise((resolve, reject) => {
    const worker = new Worker(new URL("./worker.js", import.meta.url), {
      workerData: input,
    });
    worker.on("message", resolve);
    worker.on("error", reject);
    worker.on("exit", (code) => {
      if (code !== 0) reject(new Error(`Worker exited with code ${code}`));
    });
  });
}

const result = await runWorker({ n: 42 });
console.log(result);
```

`worker.js`：

```js
import { parentPort, workerData } from "node:worker_threads";

function fib(n) {
  return n <= 1 ? n : fib(n - 1) + fib(n - 2);
}

parentPort.postMessage({ n: workerData.n, fib: fib(workerData.n) });
```

直觉：

> 主线程继续接请求，CPU 重活交给 worker。

### 10.2 Cluster：多进程模式承载更多连接

Cluster 是 Node 的多进程模式：

- 主进程 fork 多个 worker 进程
- 多个进程共享监听端口

它更像“多家分店一起接客”，适合：

- 用满多核
- 提高吞吐
- 进程级隔离

## Part 11：池（Pool）：线程池 / 对象池 / 连接池（核心是“控制资源上限”）🧠

你提纲里的三种池，背后是同一个思想：

> 资源很贵，不能无限创建；复用 + 上限 + 排队。

### 11.1 线程池

- 限制线程数量
- 避免频繁创建销毁

### 11.2 对象池

- 避免频繁创建大对象（高 GC 压力）
- 适合非常特定的性能场景

### 11.3 连接池

- 数据库连接很贵
- 必须限制并复用

手绘统一理解：

```text
请求来了
  |
  v
[Pool]
  | 有空闲资源 -> 借
  | 没空闲资源 -> 排队/拒绝/超时
  v
执行任务
  |
  v
归还资源
```

---

## Part 12：开发实践：怎么写“不会把系统搞崩”的并发代码？🧯

### 12.1 别无脑 Promise.all（它可能是 DDoS 自己）

很多新手写：

```js
await Promise.all(items.map((item) => fetchSomething(item)));
```

当 `items` 很大时，你相当于：

- 同时打爆数据库连接池
- 同时打爆下游 API
- 同时把自己内存撑爆

正确思路：限制并发（concurrency limit）。

一个简单并发控制的“队列”示意：

```text
任务进来 -> 进入队列 -> 同时只跑 N 个 -> 跑完再取下一个
```

### 12.2 背压（Backpressure）：系统说“我吃不下了”

并发控制本质上就是背压：

- 下游慢 -> 上游别再狂塞
- 保护系统稳定性

这在消息队列、流处理、HTTP 网关限流里都存在同一种思想。

## 加餐 1：并发最常见的 bug 长什么样？（竞态、死锁、饥饿）🐛

你不需要一上来背“并发理论”，你要先能识别事故形态。

### 1) 竞态条件（Race Condition）：顺序一变，结果就变

典型场景：两个任务同时修改同一个共享状态。

手绘：

```text
共享变量 balance = 100

任务A：读 balance=100 -> 准备写 80
任务B：读 balance=100 -> 准备写 50

如果没有同步：
最后可能是 80，也可能是 50（丢更新）
```

竞态最可怕的点：

> 你本地跑不出来，线上偶尔出现，还不好复现。😵

### 2) 死锁（Deadlock）：你等我，我等你，大家一起僵住

```text
任务A：拿锁1 -> 等锁2
任务B：拿锁2 -> 等锁1
=> 死锁
```

### 3) 饥饿（Starvation）：有的人永远排不到队

比如：

- 高优先级任务一直插队
- 某类请求永远得不到资源

你会看到“某些用户一直很慢”，但整体指标看起来还行。

---

## 加餐 2：同步原语（Mutex/Semaphore/Queue）到底是什么？🔧

这里用大白话讲三个你在工程里最常用的：

### 1) 互斥锁（Mutex）

语义：

> 同一时间只允许一个人进入临界区。

手绘：

```text
临界区 = 修改共享数据的代码

拿到锁 -> 进临界区 -> 出来 -> 释放锁
```

### 2) 信号量（Semaphore）

语义：

> 同一时间允许 N 个人进入（N 个名额）。

这非常像“限制并发”的思想本体。

### 3) 队列（Queue）

语义：

> 你先排队，我按顺序处理。

并发控制很多时候不是“更快”，而是“更稳”：

- 控制资源上限
- 避免下游被打爆

---

## 加餐 3：Node 的事件循环细节：为什么 `setTimeout` 不准？为什么 Promise 更“插队”？⏱️

你不需要背每个阶段，但你需要两个直觉：

### 1) 微任务（microtask）会“插队”

`Promise.then` 属于微任务，它经常会在下一次事件循环阶段之前尽快执行。

大白话：

> Promise 回调往往比 `setTimeout(fn, 0)` 更先被执行。

### 2) 定时器不是“精确闹钟”

`setTimeout(fn, 100)` 的意思更像：

> 100ms 之后“尽快”执行，但如果主线程在忙，它还是得等。

所以 event loop 卡住时：

- 定时器会延后
- 所有回调都延后

---

## 加餐 4：怎么判断 event loop 被卡住了？（线上非常好用）📉

你可以用一个很实用的指标：event loop lag（事件循环延迟）。

直觉：

- 正常：延迟很小
- 卡住：延迟突然变大

一个最小“测 lag”示意（思路级别）：

```text
每隔 100ms 记录一次“应该触发的时间”
实际触发时看偏差
偏差越大，说明主线程越忙/越堵
```

这类指标能帮你快速判断：

> 慢是因为 DB/网络，还是因为你把 CPU 重活塞进了主线程。

---

## 加餐 5：并发限制的“最小可用实现”（别只会 Promise.all）🚦

你前面看到“别无脑 Promise.all”，这里给你一个最小并发限制实现（不依赖第三方库）。

```js
export async function mapWithConcurrency(items, concurrency, worker) {
  const results = new Array(items.length);
  let nextIndex = 0;

  async function run() {
    while (true) {
      const current = nextIndex;
      if (current >= items.length) return;
      nextIndex += 1;
      results[current] = await worker(items[current], current);
    }
  }

  const runners = Array.from({ length: concurrency }, () => run());
  await Promise.all(runners);
  return results;
}
```

你以后看到“批量处理任务”时，就可以第一反应：

> 先限制并发，再谈性能。

## 加餐 6：Worker 不是只开一个：要的是“worker 池”🧑‍🏭

现实里 CPU 重活不是偶尔一次，而可能是持续存在的（例如图片压缩/加密/AI 推理前后处理）。

这时候你通常不想：

- 每个任务都 new 一个 Worker（创建成本）

你想要的是：

> 固定数量的 worker 复用，任务排队分发。

手绘：

```text
任务队列 -> Worker1
       -> Worker2
       -> Worker3
```

这和“线程池/连接池”的思想完全一致：

- 资源复用
- 控制上限
- 排队与超时

---

## 加餐 7：背压（Backpressure）在 Node 里最常见的形态：Streams 🌊

很多时候你处理的是“流”：

- 上传文件
- 下载文件
- 大 JSON/大日志

如果你一口气把所有数据读进内存：

- 内存炸
- GC 抖
- 系统卡

流式处理的核心就是背压：

> 下游处理不过来，上游就别再猛灌。

手绘：

```text
Readable -> Transform -> Writable
   |
   +-- 如果 Writable 处理慢，Readable 就要暂停/等待
```

---

## 加餐 8：超时与取消（并发系统的“刹车”）🛑

并发系统里，没有超时就等于：

> 任何一次下游卡住，都可能无限占用资源。

你需要建立两个习惯：

1. 对外部依赖设置超时（DB、HTTP、队列）
2. 对内部队列设置最大等待时间（排队也要有尽头）

否则你会出现：

- 连接池被占满
- 请求堆积
- 雪崩式扩散

---

## 加餐 9：并发不是越大越好：你要看“最短板”🪵

你可以把一个请求链路想成一条水管：

```text
Nginx 限流 -> 应用并发 -> 线程/worker -> DB 连接池 -> DB 锁/IO
```

你把应用并发从 100 提到 1000，可能不会更快，只会让：

- DB 连接池打满
- 锁冲突变严重
- p99 延迟暴涨

并发调优的本质是：

> 找到系统最短板，控制上游的注水量。

## AI 时代你最该掌握的并发能力（上帝视角版）🧠

1. **任务分类**：CPU 密集 vs IO 密集，决定用什么模型
2. **运行时模型**：Node 的 event loop + libuv；线程/进程的代价
3. **资源上限**：池、队列、超时、限流是一个思想体系
4. **把重活挪走**：worker/队列/离线任务，别堵主线程
5. **可观测性**：并发问题往往需要指标/日志/追踪来定位，而不是“看代码猜”
