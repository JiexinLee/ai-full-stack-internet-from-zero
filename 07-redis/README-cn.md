# 第七章：Redis 与缓存（重点不是 Redis，是“为什么需要缓存”）🧊

> 你的 `outline.md` 写得非常对：Redis 不是重点。  
> 重点是：为什么会出现缓存、什么时候该缓存、什么时候千万别缓存。  
> AI 可以帮你写出一段“看起来能跑”的缓存代码，但缓存真正的坑在系统层面：一致性、雪崩、击穿、穿透、热 key、容量与回源策略。

本章会按你给的顺序走：

`Memory -> Redis -> Distributed Cache -> 缓存穿透/击穿/雪崩 -> 一致性/淘汰 -> LRU/LFU/TTL -> Pipeline/Lua/事务 -> 什么时候不能/必须缓存 -> 缓存 DB/接口/HTML`

---

## 按三个问题拆解

### 1) 为什么会出现缓存？🤔

因为现实世界里，“每次都去数据库/远端服务拿数据”太贵了：

- 慢：数据库/磁盘/网络都有延迟
- 贵：数据库连接和 CPU 很宝贵
- 不稳：流量一波峰值就把后端打爆

缓存的本质是一个非常朴素的愿望：

> **同样的问题别重复算，同样的数据别重复查。**

你可以把缓存理解成“把常用的东西放手边”：

- 数据库像档案室：靠谱但远
- 缓存像桌面便签：快但不一定永久

### 2) 为什么选择 Redis？有没有别的选择？🧰

缓存这件事分很多层，Redis 只是其中很重要的一层。

你可以从“离用户越近越快”的视角看缓存层级：

```text
浏览器缓存（最快，离用户最近）
   |
CDN 缓存
   |
Nginx/反向代理缓存
   |
应用内存缓存（单机）
   |
Redis（分布式缓存）
   |
数据库（最终真相）
```

Redis 常见优点：

- 内存数据结构丰富（string/hash/list/set/zset）
- 性能高（内存）
- 支持分布式共享（多实例应用都能读同一份缓存）
- 支持过期（TTL）、淘汰策略、脚本原子性

替代方案：

- 应用内存缓存：快，但只对单机有效，扩容后不共享
- Memcached：简单 KV 缓存
- 本地缓存 + 分布式一致性方案：更复杂

结论大白话：

> Redis 是“工程上性价比很高”的分布式缓存组件，所以它很常见。

### 3) 缓存到底怎么工作？怎么用？🔍

从开发者视角，缓存设计的核心就是：

- **缓存什么？**
- **怎么失效？**
- **怎么保证一致性在可接受范围内？**
- **遇到峰值/异常时怎么保护后端？**

下面开始上主菜。

## Part 0：先建立两个重要边界 🧠

### 0.1 缓存不是事实来源（Source of Truth）

绝大多数业务里：

- 数据库：最终真相
- 缓存：加速层

你可以丢缓存，但不能丢数据库。

### 0.2 缓存是“用一致性换性能”的一种手段

缓存最常见的设计思想是：

> **允许短时间不一致（最终一致），换取整体性能与稳定性。**

当然，有些业务不能这么玩（比如钱、库存），后面会讲“什么时候不能缓存”。

## Part 1：从 Memory 开始理解缓存（为什么快）⚡

你不用背具体数字，但需要一个数量级直觉：

```text
CPU/内存：纳秒~微秒级
网络请求：毫秒级（还可能抖）
磁盘：毫秒级
```

所以把数据从“远处”搬到“近处”会带来巨大收益。

缓存就是不断做这件事：

> 把热点数据搬近一点。

---

## Part 2：Redis 能做什么？（别从命令开始，从场景开始）🎯

Redis 的常见使用场景（按你未来最可能用到的排序）：

1. 缓存热点读（商品信息、配置、公开内容）
2. 会话/登录态（session 存储）
3. 计数器/限流（访问次数、验证码次数）
4. 排行榜（zset）
5. 分布式锁（谨慎使用，理解语义）
6. 队列/异步任务（list/stream，视场景）

你会发现：  
它不是“只能当缓存”，它更像一个高性能的内存数据结构工具箱。

---

## Part 3：分布式缓存（Distributed Cache）到底解决了什么？🌍

如果你只用应用内存缓存，会出现一个尴尬问题：

```text
用户请求打到 App1：缓存命中 ✅
用户请求打到 App2：缓存没命中 ❌（又去查 DB）
```

也就是说：缓存碎成了好几份，命中率下降，DB 还是被打。

分布式缓存（Redis）解决的是：

> **多台应用共享同一份缓存。**

手绘图：

```text
            +-------> App1 ----+
用户请求 --->|                 |----> Redis（共享缓存）
            +-------> App2 ----+
                               |
                               v
                              DB（真相）
```

## Part 4：最经典的缓存模式：Cache Aside（旁路缓存）🛣️

### 4.1 读：先查缓存，没命中再查 DB，然后回填

```text
请求来了
  |
  v
查 Redis
  | 命中 -> 返回
  |
  | 未命中 -> 查 DB -> 写入 Redis -> 返回
```

### 4.2 写：先写 DB，再删缓存（或更新缓存）

写入时最常见的做法是：

1. 更新数据库（真相）
2. 删除缓存（让下次读时回源并回填）

大白话原因：

> 写缓存很容易写错（多并发下覆盖），删缓存更安全。

### 4.3 Node 伪代码（不绑定具体库）

```js
async function getUser(userId) {
  const cacheKey = `user:${userId}`;
  const cached = await redis.get(cacheKey);
  if (cached) return JSON.parse(cached);

  const user = await db.queryOne("SELECT id, email FROM users WHERE id = ?", [
    userId,
  ]);
  await redis.set(cacheKey, JSON.stringify(user), { EX: 60 });
  return user;
}

async function updateUserEmail(userId, email) {
  await db.exec("UPDATE users SET email = ? WHERE id = ?", [email, userId]);
  await redis.del(`user:${userId}`);
}
```

注意：这段代码的“重点”不是 API 调用，而是顺序和思路。

## Part 5：缓存穿透 / 击穿 / 雪崩（三大噩梦）👻

这三个词你不需要背定义，你需要背“事故长什么样”。

### 5.1 缓存穿透：请求的 key 根本不存在

现象：

- 攻击者或 bug 一直请求不存在的 id（比如 `user:-1`）
- Redis 永远 miss
- 请求全部打到 DB

手绘图：

```text
请求 user:-1
   |
   v
Redis：没有
   |
   v
DB：也没有（但你还是被迫查了）
   |
   v
每次都回源 -> DB 被打爆
```

常见治理：

- 参数校验（最便宜）
- 缓存空值（短 TTL）
- 布隆过滤器（Bloom Filter）提前判断“可能存在/一定不存在”

### 5.2 缓存击穿：一个“热点 key”刚好过期

现象：

- 某个超级热门 key（比如首页配置、爆款商品）
- 刚好过期
- 同一瞬间大量请求一起 miss
- 一起打 DB（惊群）

治理思路：

- 热点 key 互斥重建（singleflight / mutex）
- 过期时间加随机抖动（jitter）
- 预热（warmup）

### 5.3 缓存雪崩：大量 key 同时过期或缓存层故障

现象：

- 大量 key 同一时间过期（比如都设置了 60s）
- 或 Redis 故障
- 大量请求全部打 DB，DB 也扛不住

治理思路：

- TTL 加随机抖动（最常见）
- 多级缓存（本地 + Redis）
- 降级策略（返回兜底数据/提示）
- 限流（入口层保护）

## Part 6：缓存一致性：你到底要“多一致”？⚖️

缓存一致性没有银弹，只有取舍。

你要先问业务一个问题：

> 这个数据允许“短时间旧一点”吗？

### 6.1 常见一致性目标

- 强一致：几乎不允许旧数据（钱、库存、权限）
- 最终一致：允许短时间旧（商品详情、配置、公开信息）

### 6.2 最常见策略：更新 DB 后删除缓存

这是工程里最常用、最稳妥的一类。

### 6.3 你要知道的两个真实坑

1. **删除缓存失败怎么办？**
   - 需要重试/补偿（消息队列、后台任务）
2. **并发写导致旧数据回填怎么办？**
   - 需要版本号/时间戳/写入保护（更高级）

---

## Part 7：缓存淘汰：LRU / LFU / TTL（别背算法，记直觉）🧹

Redis 内存不是无限的，所以需要淘汰策略。

### 7.1 TTL（过期时间）

你告诉 Redis：

> 这个 key 只能活 60 秒。

优点：简单  
缺点：不是按“热度”淘汰

### 7.2 LRU（最近最少使用）

直觉：

> 最近没用过的，先丢掉。

### 7.3 LFU（最不经常使用）

直觉：

> 用得少的，先丢掉。

### 7.4 你作为开发者要记住的重点

淘汰策略不是背概念，而是要知道：

- 你的缓存是“容量型”还是“时间型”
- 内存满了会发生什么（谁被淘汰）
- 被淘汰会不会导致回源风暴

---

## Part 8：Pipeline：为什么它能让 Redis 更快？🚄

Redis 很快，但网络往返（RTT）也很贵。

Pipeline 的大白话：

> 别一问一答，先把 100 个问题一次性发过去，再一起收答案。

手绘图：

```text
不使用 pipeline：
请求1 -> 响应1 -> 请求2 -> 响应2 -> ...（来回跑很多趟）

使用 pipeline：
请求1..请求100 一次发过去 -> 响应1..响应100 一次收回来
```

这能显著减少网络往返次数。

## Part 9：Lua：为什么它是 Redis 的“原子操作外挂”？🧠

Redis 单条命令是原子的，但业务常常不是单条命令能解决。

比如“计数 + 过期”：

- 如果你分两条命令写，很容易出现并发时序问题

Lua 脚本让你能在 Redis 内部一次性完成多个操作，保证原子性。

一个经典脚本（限流雏形）：

```lua
-- KEYS[1] = key
-- ARGV[1] = limit
-- ARGV[2] = window_seconds

local current = redis.call("INCR", KEYS[1])
if current == 1 then
  redis.call("EXPIRE", KEYS[1], ARGV[2])
end
if current > tonumber(ARGV[1]) then
  return 0
end
return 1
```

直觉：

> Lua 是把“多步操作”打包成“一步原子操作”。

---

## Part 10：Redis 事务（MULTI/EXEC）要怎么理解？🧾

Redis 的事务和数据库事务不是一回事。

大白话：

- `MULTI` 开始排队
- 命令进入队列
- `EXEC` 一次执行队列

它更像“批处理 + 一次性执行”，并不等同于数据库那种 ACID 事务。

如果你需要“严格原子”的多步逻辑，Lua 往往更直接。

## Part 11：什么时候不能缓存？什么时候必须缓存？🚦

这是你 `outline.md` 里“更高层”的重点。

### 11.1 什么时候不能缓存（或非常谨慎）

- 钱、库存、权限这类强一致需求
- 用户私有、敏感数据（怕泄露/串号）
- 强实时状态（秒级变化、需要绝对准确）
- 写多读少的数据（缓存收益不大，反而一致性成本高）

### 11.2 什么时候必须缓存

- 读多写少，且热点明显（商品详情、配置）
- 下游很贵（数据库很重、第三方接口贵）
- 流量波动大，需要削峰（配合 TTL 和降级）

### 11.3 什么时候缓存数据库？什么时候缓存接口？什么时候缓存 HTML？

你可以用“缓存位置”来理解：

- 缓存数据库查询结果：最常见（Cache Aside）
- 缓存接口响应：适合公共内容、读多写少接口
- 缓存 HTML：适合公开页面（可以到 CDN/Nginx 层）

手绘图（越往上越靠近用户）：

```text
HTML 缓存（CDN/Nginx）  -> 最靠近用户
接口缓存（Nginx/应用/Redis）
DB 查询缓存（Redis）
DB（真相）             -> 最可靠但最贵
```

## 加餐 1：Key 设计（Redis 用起来顺不顺，全看你 Key 写得像不像人）🗝️

新手最容易把 Redis 用成“全靠运气”的原因就是：Key 设计乱。

### 1) Key 命名三件套：命名空间 / 主键 / 版本号

推荐你从第一天就养成这种 Key 风格：

```text
{namespace}:{entity}:{id}:v{version}
```

例子：

```text
user:profile:123:v1
product:detail:888:v2
order:summary:10001:v1
```

为什么要版本号？

因为你迟早会遇到：

> “缓存结构升级了，但旧缓存还在，解析报错/字段缺失/逻辑错乱。” 😵

有版本号你就能：

- 升级时改成 v2
- 旧 key 自然过期或批量清掉

### 2) Key 的粒度：别太粗，也别太碎

- 太粗：一个 key 放一坨巨大 JSON，变成 big key（后面讲）
- 太碎：一个页面需要读 30 个 key，网络往返变多

经验：

- 展示类数据可稍“粗”一点（减少 round trips）
- 强一致类数据别在 Redis 里拼花活

### 3) 序列化：JSON 不是原罪，但别“无脑 JSON”

常见做法：

- 读多写少：JSON OK
- 字段经常单独更新：考虑 hash（按字段更新）

手绘理解：

```text
JSON：整体读写
Hash：字段级读写
```

## 加餐 2：TTL 策略（最常见的事故起点）⏱️

### 1) TTL 不要整齐划一

如果你给大量 key 都设成 `60s`：

- 你很容易制造“雪崩”

正确姿势：加随机抖动（jitter）

```text
ttl = base + random(0..base*0.1)
```

### 2) TTL 不只是“多久过期”，也是“多久能接受旧数据”

你应该用业务语言定义 TTL：

- 配置：允许旧 1 分钟吗？
- 商品详情：允许旧 5 分钟吗？
- 订单状态：允许旧 1 秒吗？（很多时候不行）

### 3) 逻辑过期（Logical Expiration）解决击穿的一个套路

思路：

- key 永不过期（或很久）
- value 里带一个 `expireAt`
- 读到过期数据仍可先返回旧值（可选），同时后台异步重建

这是“高可用优先”思路：

> 宁愿短时间旧一点，也别把 DB 打爆。

## 加餐 3：Hot Key / Big Key（Redis 线上最常见的两类大坑）🔥🪨

### 1) Hot Key：一个 key 被疯狂访问

现象：

- Redis CPU 飙高，但不是全局，是某个 key 特别热
- 下游 DB 可能也被同一热点牵连

常见原因：

- 首页配置
- 爆款商品
- 热门直播间

治理方向：

- 本地缓存 + Redis（两级）
- 热点拆分（把一个 key 拆成多个分片 key，再在应用层合并）
- 预热 + 互斥重建
- CDN/Nginx 上移缓存（能靠近用户就靠近）

### 2) Big Key：一个 key 太大

现象：

- 单次读写耗时变大
- 可能导致阻塞（Redis 单线程执行命令）
- 网络传输成本大
- 复制/AOF 压力大

治理方向：

- 拆分结构（hash/list 分片）
- 只存必要字段（别把整页 HTML/大对象全塞）
- 给大 key 设置合理 TTL，避免永久占内存

## 加餐 4：Redis 不是“永远不会丢数据”的（持久化与高可用你要有概念）🧯

Redis 常见两件事你必须知道：

### 1) 持久化（Persistence）

Redis 是内存系统，但可以做持久化：

- RDB：定期快照
- AOF：追加日志（更接近“记录每一步写操作”）

大白话：

- RDB：拍照存档
- AOF：写流水账

你要有直觉：

- 如果 Redis 挂了，是否允许缓存丢失？（大多数缓存允许）
- 如果 Redis 被当作“队列/状态存储”，那就要非常谨慎

### 2) 高可用（HA）

常见形态：

- 主从复制：提升读、提供冗余
- 哨兵：自动故障转移
- 集群：分片扩容 + 部分故障隔离

你不需要背命令，但要知道它们在解决什么：

> “一台 Redis 不够用/不够稳”，就会往这些方向演进。

---

## 加餐 5：分布式锁（最容易写错的功能之一）🔒

Redis 分布式锁常见写法是：

- `SET lock_key value NX PX 30000`

但“能写出来”不代表“语义正确”。

你至少要知道这些坑：

1. 锁要有过期时间（防止死锁）
2. 解锁要校验 value（防止删掉别人的锁）
3. 业务需要幂等与重试策略（锁不是万能）

大白话：

> 分布式锁不是“我加了个 lock 就安全了”，它是“我给自己增加了一个必须正确实现的新系统”。😅

如果只是防止同一用户重复提交，很多时候：

- 业务幂等 token
- 数据库唯一约束

比分布式锁更稳。

---

## 加餐 6：缓存与安全（最容易被忽略，但很致命）🧨

你必须建立两个安全直觉：

1. **别把敏感数据直接塞缓存**（尤其是用户私密信息）
2. **别让不同用户共用同一个 key**（串号=事故）

常见事故长这样：

- key 没带 userId，导致 A 用户读到 B 用户数据

所以你会看到很多缓存 key 都带：

```text
user:{userId}:...
tenant:{tenantId}:...
```
